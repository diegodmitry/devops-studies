# Dummy Systemd Service

A long-running `systemd` service that runs in the background and writes a message to a log file every 10 seconds.

This project is part of my DevOps learning journey, following the roadmap.sh DevOps path.

- Roadmap.sh project: https://roadmap.sh/projects/dummy-systemd-service
- Level: Beginner
- Main topics: Linux, Bash, systemd, Services, Logs, Process Management

## Goal

The goal of this project is to create a custom `systemd` service called `dummy.service` that starts a Bash script in the background and keeps it running.

The script simulates a simple background application by writing a message to a log file every 10 seconds.

This project helps practice how to:

- Create a Bash script for a long-running process
- Create a custom `systemd` unit file
- Start and stop a service manually
- Enable and disable a service at boot
- Check the service status
- Follow logs generated by the service
- Understand how `systemd` manages background processes

## Implemented features

This project currently:

- Creates a Bash script called `dummy.sh`
- Runs an infinite loop using `while true`
- Writes the message `Dummy service is running...` to `/var/log/dummy-service.log`
- Waits 10 seconds between each log entry
- Creates a custom `systemd` service called `dummy.service`
- Uses `ExecStart` to run the script through `systemd`
- Uses `Restart=always` so the service restarts if it fails
- Uses `WantedBy=multi-user.target` so the service can start on boot when enabled
- Was successfully enabled with `systemctl enable`
- Was successfully started with `systemctl start`
- Was verified as active and running with `systemctl status`
- Was verified by following the log file with `tail -f`

## Script

The script is located at:

```text
/home/{$USER}/dummy-systemd-service/dummy.sh
```

Script content:

```bash
#!/usr/bin/env bash

while true; do
  echo "Dummy service is running..." >> /var/log/dummy-service.log
  sleep 10
done
```

The script keeps running forever and appends a new line to the log file every 10 seconds.

## systemd service file

The service file is located at:

```text
/etc/systemd/system/dummy.service
```

Service content:

```ini
[Unit]
Description=Create a long-running systemd service that logs to a file

[Service]
ExecStart=/home/{$USER}/dummy-systemd-service/dummy.sh
Restart=always

[Install]
WantedBy=multi-user.target
```

## How to run

Go to the project folder:

```bash
cd /home/{$USER}/dummy-systemd-service
```

Give execution permission to the script:

```bash
chmod +x dummy.sh
```

Create or copy the service file to the `systemd` directory:

```bash
sudo cp dummy.service /etc/systemd/system/dummy.service
```

Reload `systemd` after creating or changing the service file:

```bash
sudo systemctl daemon-reload
```

Start the service:

```bash
sudo systemctl start dummy
```

Enable the service to start automatically on boot:

```bash
sudo systemctl enable dummy
```

## Checking logs

Follow the application log file:

```bash
sudo tail -f /var/log/dummy-service.log
```

Follow the `systemd` journal for the service:

```bash
sudo journalctl -u dummy -f
```

> Note: the script writes the application message directly to `/var/log/dummy-service.log`. Because of that, `tail -f` shows the application messages. `journalctl` is still useful for checking service lifecycle events, start failures, stop events, restart attempts, and other `systemd` messages.

## How it works

The Bash script runs an infinite loop:

1. It writes a message to `/var/log/dummy-service.log`
2. It waits 10 seconds
3. It repeats the same process forever

The `systemd` service starts this script using the `ExecStart` directive.

The `Restart=always` option tells `systemd` to restart the service automatically if it stops unexpectedly.

The `WantedBy=multi-user.target` option allows the service to be enabled at boot for a normal multi-user Linux environment.

- `bash`

- `tail`
- `chmod`

The service writes to `/var/log/dummy-service.log`, so it needs permission to write to `/var/log`.

When no `User=` is defined in the service file, the service runs as root by default.

## Current limitations

- The script runs forever until the service is stopped
- The log file path is fixed as `/var/log/dummy-service.log`
- The log message is fixed as `Dummy service is running...`
- The sleep interval is fixed at 10 seconds
- The service does not define a dedicated non-root user
- The service does not define `RestartSec`
- The project does not include log rotation yet
- Application logs are written directly to a file, not to stdout
- The script header comments should be updated to fully match this project name and purpose
